# Action Rules for Tasks
* **작업 수립은 영어로 진행하고 최종 결과만 한글로 작성**
* **새로운 작업 요청으로 인해 논리적인 위배가 발생하는 경우, 내용을 사용자에게 설명하고 확인 받아야 함**

# 주요 규칙
* **모든 작업전에 작업 요청 문서를 우선 작성하고 확인을 받아야함**
* **작업의 단계가 구분되어 있는 경우 단계별 진행을 마칠때 마다 작업 항목 체크표시**
* .md와 같은 문서화 작업은 요청한 것만 작업해야함
* 불필요한 문서화 작업은 하지 말 것

# UI 작업 규칙
* **Flutter 화면(Windows Desktop / Web) 또는 웹 화면을 수정하거나 새로 만드는 작업의 경우 작업 요청 문서에 반드시 ASCII 다이어그램을 포함**
  * 스크린 레이아웃, 위젯 배치, 탭 구조 등을 ASCII로 시각화
  * 구현 전 의도한 화면 구성을 명확히 전달하고 확인받기 위함

# 요청 작업을 수행하는 방법 (다음의 단계로 수행)
* 작업 요청의 계획을 수립
* 계획의 검증 및 승인
* 작업 수행
  * 작업 요청의 단계를 완료하면 완료 항목에 체크표시

# docs 폴더 규칙
## docs/requests 작업 요청 기록
* 작업 요청 기록은 `/docs/requests/YYYY-MM/` 월별 하위 폴더에 저장
* 파일명: `YYYYMMDD-HHmmss-설명.md` (타임스탬프 + kebab-case 설명)
  * 예: `20260327-143052-panel-width-fix.md`
  * **타임스탬프는 반드시 `date +"%Y%m%d-%H%M%S"` 명령어로 현재 시각을 조회하여 사용** (추정/하드코딩 금지)
  * 타임스탬프는 파일 생성 시점 기준 (충돌 방지)
  * Agent의 개별 작업으로 생성되는 경우 `YYYYMMDD-HHmmss-{Agent-Name}-설명.md` (타임스탬프 + kebab-case 설명)로 파일명 생성
    * 예: `20260327-143052-frontend-claude-panel-width-fix.md`
    * 타임스탬프를 동일하게 유지하여 같은 작업 요청임을 명확히 함
* 개별 작업 요청은 `docs/requests/TEMPLATE.md`의 형식을 따름
  * 상단 메타데이터 테이블 필수 (유형, 영역, 날짜, 상태, 관련)
  * 필요한 섹션만 작성 (불필요한 섹션 강제 금지)
* 작업 결과는 수행계획을 항목별로 정리
  * 수행계획을 완료한 항목은 체크 표시
  * 수행계획을 완료하지 못한 항목은 체크 표시하지 않음
* **버그 수정(fix) 유형의 작업은 작업 결과에 다음 분석 섹션을 포함해야 함** (해당 없는 항목은 생략 가능)
  * Root cause: 문제가 왜 발생했는지
  * Change: 실제로 어떤 수정이 들어갔는지
  * Recurrence prevention: 같은 문제가 반복되지 않게 하는 방법
  * Verification: 어떻게 검증했고, 결과가 무엇인지
  * Remaining risk: 아직 확인되지 않은 부분
* **작업 완료 시 인덱스 갱신 필수**
  1. **월별 인덱스** (`docs/requests/YYYY-MM/INDEX.md`): 상세 테이블에 항목 추가/갱신 (최신 항목을 테이블 상단에 추가)
     * 테이블 컬럼: `파일 | 제목 | 유형 | 영역 | 상태 | 관련`
     * 파일 링크는 같은 폴더 내 상대경로 (예: `[파일명](파일명.md)`)
     * 월별 INDEX가 없으면 새로 생성
  2. **루트 인덱스** (`docs/requests/INDEX.md`): 새 월이 시작되면 월별 인덱스 테이블에 행 추가
     * 루트 INDEX는 월별 요약만 유지 (개별 작업 항목을 기록하지 않음)
     * 테이블 컬럼: `월 | 건수 | 주요 작업`
* **새 세션 시작 시 `docs/requests/INDEX.md`(루트)를 읽어 과거 작업 맥락 파악**
  * 유사 작업 검토가 필요한 경우 해당 월의 `docs/requests/YYYY-MM/INDEX.md`를 추가로 읽어 상세 정보(영역, 상태, 관련) 확인
